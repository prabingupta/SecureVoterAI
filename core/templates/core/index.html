About the Platform

Built Specifically for
Islington College Elections

SecureVoter AI is a next-generation digital election platform developed from the ground up for Islington College, Kamal Pokhari. It replaces unreliable paper-based ballots with an AI-powered, multi-layered security system that ensures every vote is cast by a verified student — and counted exactly once.

Security Infrastructure

8 Layers of Fraud Protection

Every vote passes through a comprehensive stack of AI-driven security checks, cryptographic protections, and database-level constraints before it is recorded.

01

AI Face Verification

Encrypted 128-dimensional biometric face encoding maps each student's unique facial geometry. The model compares live captures against stored embeddings with sub-millisecond precision.

DeepFace · dlib · FaceNet
02

Real-Time Liveness Detection

Active gesture challenges (blink, head turn) prevent photo, video, and mask-based spoofing attacks. No static image can pass the liveness gate.

Gesture Challenge · Anti-Spoof
03

JWT Token Authentication

Every session is secured with signed JSON Web Tokens. Tokens expire automatically and are blacklisted on logout, blocking session replay attacks.

JWT · Token Blacklist · HTTPS Only
04

Append-Only Vote Storage

Votes are written to an immutable append-only table. Database-level unique constraints on (voter_id, position_id) enforce a strict one-vote-per-position rule.

PostgreSQL · Unique Constraints
05

Rate Limiting

All API endpoints are rate-limited per IP and per account. Brute-force attempts on login and face verification trigger automatic progressive cooldowns and account lockout.

Django Ratelimit · IP Throttle
06

Account Lockout

Consecutive failed authentication attempts lock the account for a configurable cooldown period, protecting against automated credential stuffing.

Progressive Lockout · Admin Alerts
07

Behavioral Fraud Analysis

The system tracks login patterns, geographic anomalies, and device fingerprints. Suspicious behavior triggers alerts to the election administrator in real time.

IP Tracking · Anomaly Detection
08

Full Audit Logging

Every significant action — login, face scan, vote submission, admin change — is timestamped and written to a tamper-evident audit log, ensuring complete election accountability.

Immutable Logs · Admin Dashboard
The Process

How the Election Process Works

From registration to results — every step is transparent, verified, and secured by multiple independent mechanisms.

01

Student Registration

Students visit the SecureVoter AI portal and complete a three-step registration: personal details (name, Student ID, department, year), face enrollment (live biometric scan via webcam), and a final review & submission.

  • Student ID validated (ISL-XXXX format)
  • Department verified against college records
  • Biometric face data encrypted and stored
Portal → Register
02

Admin Verification & Approval

The IT Administration team reviews each submitted registration. Student IDs are cross-checked against the official Islington College enrollment database. Suspicious or incomplete registrations are flagged and rejected before approval is granted.

  • ID cross-checked with enrollment records
  • Face photo quality assessed
  • Approval email sent to verified students
Admin Dashboard → Approve
03

Secure Login + Live Face Match

On voting day, students log in with their Student ID and password. After credential verification, they complete a live face scan with a liveness challenge (blink detection). The system compares the live capture to the registered biometric embedding in real time.

  • Password hashed with bcrypt/Argon2
  • Live blink challenge defeats spoofing
  • Face similarity checked (>85% threshold)
  • JWT session token issued on success
Login → Face Scan → Session
04

Cast Your Vote

Verified students access the live election dashboard and view all candidates for each position. They select one candidate per position and confirm their ballot. The system atomically writes the vote and immediately marks the position as voted for that student.

  • One vote per position enforced at DB level
  • Atomic transaction prevents partial submissions
  • Vote receipt generated (no candidate linkage)
  • Audit log entry created immediately
Election Portal → Cast Vote
05

Tally, Audit & Results

After the voting period closes, administrators run the verified tally. Results are computed from the append-only vote table. The full audit log is reviewed for anomalies before results are officially published on the portal for all students.

  • Votes tallied from immutable record
  • Audit log reviewed for irregularities
  • Results published on portal
  • Winners notified via official email
Admin → Publish Results

Who Can Vote

Eligibility & Voting Rules

Who Is Eligible

  • Currently enrolled students at Islington College, Kathmandu
  • Students from Computing, Artificial Intelligence, or Networking departments
  • Students who have completed the SecureVoter AI registration process
  • Students whose registration has been approved by IT Administration
  • Students able to complete the live face verification at time of voting

Voting Rules

  • Each student may cast exactly one vote per position — no exceptions
  • Proxy voting, vote sharing, or voting on behalf of others is strictly prohibited
  • Sharing login credentials or bypassing face verification is a violation
  • Votes cannot be changed once submitted
  • Attempting to manipulate, automate, or tamper with the voting system will result in immediate disqualification and disciplinary action

Your Vote is Completely Anonymous

SecureVoter AI records that you voted but never links your identity to your specific ballot choice. The vote tally system uses anonymised records that cannot be traced back to individual students — not even by administrators.
